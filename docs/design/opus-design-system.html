Moodarr Design System

Screening Desk — the visual language of the live Finder, documented as built

This is a descriptive design system, not a rebrand. Every token, surface, and component below is drawn from the Moodarr main page as it exists today in src/client/App.tsx and src/client/styles.css. It captures the warm, papery, catalog-console character of the app so that new surfaces — the admin experience in particular — can extend the same language without drifting into generic dashboard styling.

01

Design principles

Moodarr feels like a private screening desk for a Plex library, not a streaming app and not a SaaS dashboard. The chrome stays quiet so poster art and the watch decision carry the color. These five principles govern every screen, including admin.

1. Calm, papery, tactile

Warm off-white paper grounds the UI. A faint teal grid and soft, low-contrast shadows give surfaces a slight physicality without skeuomorphic noise. Avoid streaming darkness and avoid flat clinical white.

2. One accent at a time

Teal is the single working accent for selection and primary action. Rust is reserved for requests and caution. Plex gold and Seerr green appear only as source indicators. No purple/blue AI gradients.

3. The decision is the content

Score, availability source, runtime, and the match reason line up predictably so a user can scan twenty options fast. Chrome never competes with the poster or the ranking explanation.

4. Availability is never ambiguous

Plex and Seerr can disagree. Source state is always shown with a label and color, never color alone, and request actions only appear when they are safe to take.

5. Admin is operational, Finder is editorial

Finder shows only what a watch decision needs. Health, sync, tokens, diagnostics, and the support bundle live in admin, which is allowed to be denser and more instrument-like.

6. Quiet by default, legible under load

Numbers are tabular, metadata is pill-shaped, and skeletons mirror final layout. Secrets are always masked or redacted on screen and in any exported diagnostics.

07

Surfaces & backdrop

The page itself carries a fixed teal grid that fades from top to roughly 72% down, masking out at the bottom. This is the “catalog console” texture — subtle enough to read as paper grain, not a chart. Panels float above it with the warm shadow and inner highlight.

Panel

Translucent --panel over the grid, 1px --line border, 10px radius, shadow + inner edge. The default container for everything.

Inset / soft

An inner well using --panel-soft. Holds inputs, confirm boxes, runtime facts, and chat history.

Empty / placeholder

Dashed --line-strong border with a faint diagonal teal wash. Used for empty results and loading wells.

Layout model

The app is a centered shell (max 1480px, 18px gutter) with a persistent top bar carrying the brand lockup and icon navigation. Each view owns a two-column grid: a wide working column and a fixed-width rail.

10

Interaction states

Loading

Skeletons mirror the final layout (poster block + text lines + meta pills) with a left-to-right shimmer. Spinners use a 900ms linear rotation. Never collapse layout while loading.

Empty

A dashed placeholder well with an icon, a one-line heading, and a short prompt. Invites the next action rather than reporting a void.

Hover & focus

Cards lift 2px and gain an accent border. All interactive elements show a 2px --accent focus ring at 2px offset. Active press nudges translateY(1px) scale(.99).

Disabled & blocked

Disabled controls drop to 0.56 opacity with a not-allowed cursor. Blocked actions surface an inline reason — never a silent dead button.

Notice / error

A bordered notice strip with a warning glyph and muted text. Global notices center above content; rail notices sit inline. Errors must redact tokens and keys.

Degraded provider

When the recommendation provider is unavailable, ranking falls back to deterministic scoring and the state is reported in admin — Finder keeps working without explanation.

12

Accessibility

✓Label every icon-only control. Nav tabs, feedback thumbs, toggles, and delete buttons all carry an aria-label and a title.
✓Keep visible labels on inputs. Where space forces a hidden label, use .sr-only — never an unlabeled field.
✓Pair color with text. Availability, health, and ratings always carry a word or number, never color alone.
✓Mark live regions. The chat log and async result areas announce updates with aria-live="polite".
✓Visible focus. A 2px accent ring at 2px offset on every focusable element; toggles expose aria-pressed.
×No motion-only meaning. The breathing dot is decoration; its real state is in the adjacent text.
×No silent disabled actions. If an action is unavailable, say why inline.
×Never render real secrets. Tokens and keys are masked on screen and redacted from exports and errors.
13

Admin application guidance

Admin extends the same language but earns a denser, more instrument-like register. It is allowed more information per panel than Finder, but it must still feel like the same papery console.

Panel taxonomy

Group admin into purposeful panels, each opened by an icon + 14/760 title: Access (admin token, auth state), Integrations (Plex / Seerr / OpenAI), Health (status + library metrics + tests), Recommendation engine (diagnostics), and Runtime (facts + sync + support bundle).

Density & rhythm

Admin panels use 16–20px padding. Settings are grouped in fieldsets with a small bold legend. Inputs sit on --panel-soft wells. Status rows and metric tiles carry the operational state.

Action grouping

Destructive or outbound actions (run sync, support bundle, connection tests) are clustered and clearly labeled with verbs. Save lives at the end of its form. In-progress actions swap their icon for a spinner and disable siblings.

Secrets & trust

All credential inputs are type="password" with autocomplete="off", and show “Configured” rather than the value once set. Support-bundle copy promises credential redaction while telling users to inspect the bundle before sharing because private operational data can remain. This trust framing is part of the visual design.

Workflows admin must cover. Browser admin token storage; Plex / Seerr / OpenAI integration settings; fixture mode; sync interval; sync-Seerr toggle; review retention and max queries; health connection tests; manual sync; sync history and next-run timing; runtime facts; credential-redacted support bundle export; and recommendation-engine diagnostics. The companion opus-admin-mockup.html applies this system to all of them.
